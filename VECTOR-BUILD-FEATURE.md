# 向量构建功能更新报告

## 更新日期
2026-05-21

## 更新内容

### 1. 核心功能变更

#### 之前的流程
- 导入文件夹后，立即处理所有文件
- 生成向量后存储在内存中，刷新页面后丢失
- 无法区"已导入"和"已向量化"的文档

#### 现在的流程
1. **导入文件夹** - 只保存文件信息，不处理
   - 文档状态：`待构建`
   - 文件引用被保存，等待处理

2. **点击"构建向量"按钮** - 开始批量处理
   - 显示进度条和当前处理的文件名
   - 实时显示处理进度（x/总数）
   - 逐个文档解析、分块、向量化

3. **压缩存储** - 使用 pako (gzip) 压缩后存储到 localStorage
   - 向量数据持久化，刷新页面不丢失
   - 压缩率：约 70-80%（取决于数据）
   - 自动加载已存储的向量数据

### 4. 使用流程

```
1. 点击"导入文件夹"
   ↓
2. 选择文件夹（包含 PDF/Word/TXT/MD 文件）
   ↓
3. 文件列表显示，状态为"待构建"
   ↓
4. 点击"构建向量"按钮
   ↓
5. 显示进度条，逐个处理文档
   - 解析文档
   - 文本分块
   - 生成向量
   - 压缩存储
   ↓
6. 构建完成，显示向量数量
```

### 5. 数据存储

#### localStorage 键值
```javascript
'vaultseek-documents'      // 文档元数据
'vaultseek-vectors-compressed' // 压缩的向量数据
```

#### 压缩流程
```javascript
原始数据 (JSON) → pako.gzip() → Base64编码 → localStorage
```

#### 解压流程
```javascript
localStorage → Base64解码 → pako.inflate() → 解析 JSON → 内存 Map
```

### 6. 性能优化

| 优化项 | 说明 | 效果 |
|--------|------|------|
| gzip 压缩 | 使用 pako 压缩向量数据 | 减少 70-80% 存储空间 |
| 批量保存 | 每处理完一个文档保存一次 | 防止数据丢失 |
| 按需加载 | 初始化时加载已存储数据 | 减少冷启动时间 |
| 增量处理 | 逐个处理文档，不阻塞 UI | 更好的用户体验 |

### 7. 注意事项

1. **浏览器存储限制**
   - localStorage 通常限制 5-10MB
   - 建议：控制向量总数在 1000 以内
   - 超出限制时，旧数据会被自动清理

2. **大文件处理**
   - 单个文件超过 10MB 可能失败
   - 建议：限制单文件 5MB 以内

3. **内存使用**
   - 每个向量 1024 维 Float32 = 4KB
   - 1000 个向量 ≈ 4MB

4. **持久化限制**
   - 刷新页面后向量数据保留
   - 清空浏览器缓存会清除数据
   - Electron 版本可使用文件系统持久化
